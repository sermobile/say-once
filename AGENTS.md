# Инструкции для ИИ-агентов

Этот файл — для агентов (Claude Code, Codex, Cursor и т.п.), которые помогают автору или контрибьютору дописывать книгу `say-once`.

## О книге

Книга про передачу информации так, чтобы её не пришлось переспрашивать.
Строится **снизу вверх**: сначала реальные кейсы плохой коммуникации, потом из них выводятся принципы. Правила всегда привязаны к примерам.

Подробнее — в [`preface.md`](preface.md).

## Структура

- [`preface.md`](preface.md) — предисловие
- [`rules.md`](rules.md) — свод правил, накопленных из примеров
- [`map.md`](map.md) — карта правил: семейства, пары, архетипы, стадии, роли
- [`limits.md`](limits.md) — шесть семейств ограничений: когда правила не работают
- [`examples/`](examples/) — кейсы (один файл = один кейс)
- [`thoughts.md`](thoughts.md) — сырые мысли, связи, идеи будущих правил
- [`references.md`](references.md) — смежное чтение и сопоставления
- [`bot/`](bot/) — Telegram-бот для сбора кейсов на ходу
- [`README.md`](README.md) — двуязычная аннотация

## Язык

Книга на русском. Все примеры, правила, мысли — на русском.
README двуязычный (ru/en). Имена идентификаторов в коде — на английском.

## Рабочий цикл

1. Контрибьютор описывает ситуацию своими словами (часто из чата, разговора, опыта).
2. Агент формализует её по шаблону примера (см. ниже).
3. Агент проверяет: покрывают ли уже существующие правила этот случай.
   - Покрывают — обогащает существующее правило, добавляет ссылку на новый пример.
   - Не покрывают — формулирует новое правило.
4. Агент обновляет `rules.md`, ссылки между файлами, при необходимости — `thoughts.md`.
5. **Если добавлено или существенно изменено правило — обновить [`map.md`](map.md):**
   - вписать правило в подходящую группу раздела «По семействам»
   - если у правила есть пара «отправитель ↔ получатель» — добавить в раздел пар
   - если правило специализирует существующее — добавить в «Архетипы и специализации»
   - вписать на нужную стадию в «Жизненный цикл задачи»
   - вписать на нужный переход в «Роли»
6. Коммит и пуш — только по явному запросу контрибьютора.

## Шаблон примера

Файл создаётся в `examples/` с описательным именем в snake_case, **без числовых префиксов**.

```markdown
# Название кейса (без слова «Пример»)

## Ситуация
Кратко — кто, где, какой контекст.

## Как дали информацию
Дословно или близко к дословно — как была передана информация.

## Что было не так
Конкретно: какие усилия пришлось приложить получателю, что пошло не так.

## Как надо было
Альтернативная формулировка / способ передачи.

## Принцип
1–2 предложения — обобщение, которое потом превратится в правило.
```

## Шаблон правила в `rules.md`

```markdown
## Правило N: Краткий заголовок

Тело правила — в утвердительной форме или повелительном наклонении.
Лаконично. Без «как известно», «безусловно» и прочей воды.

**Признак нарушения:** конкретный наблюдаемый сигнал, что правило нарушено.

**Когда правило неприменимо:**
- 2–4 пункта, где правило не работает или применяется во вред

_Источник: [Название примера](examples/имя_файла.md)_
```

Если правило связано с другими — добавь блок:

```markdown
**Связь с правилом N:** ...
```

Блок «Когда правило неприменимо» — обязательный. Без него правило выглядит догмой и не выдерживает критики умного скептика: «а если…?». Каждое правило имеет границы, и они должны быть видны.

Когда формулируешь новое правило, сверься с [`limits.md`](limits.md): пункты «Когда неприменимо» обычно попадают в одно из шести семейств ограничений. Если кейс не попадает ни в одно — возможно, ты открыл новое семейство, и `limits.md` стоит дополнить.

## Тон и стиль

- Лаконично. Конкретно. Без обобщений ради обобщений.
- Реальные кейсы важнее теории.
- Не использовать эмодзи в текстах книги.
- Не писать «мы», «давайте», «вы должны» — нейтральная подача.
- Каждое правило должно иметь **проверяемый признак нарушения**.

## Этика и приватность

- Имена и фамилии реальных людей в примерах **не указывать**.
- Заменять на имена (без фамилии), роли, инициалы.
- Названия конкретных компаний/продуктов — анонимизировать, если контрибьютор не сказал иначе.

## Что НЕ делать

- Не вводить числовые префиксы в имена файлов примеров. Порядок не зашит в файловой системе — это применение правила 11 (формат как контракт) к самой книге.
- Не дублировать смысл существующих правил. Лучше уточнить/расширить уже имеющееся.
- Не выдумывать кейсы. Только реальные ситуации, рассказанные контрибьютором.
- Не править несколько правил в один проход без понимания связей между ними.
- Не делать коммит/пуш без явного запроса.
