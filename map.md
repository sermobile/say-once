# Карта правил

Правила образуют семейства. Внутри семейства они говорят про одно и то же, но в разных носителях. Между семействами — образуют пары «отправитель ↔ получатель».

---

## Пары «отправитель ↔ получатель»

Многие правила существуют парами. Одно говорит, что должен сделать отправитель, второе — что должен сделать получатель в той же ситуации.

```
   ОТПРАВИТЕЛЬ                              ПОЛУЧАТЕЛЬ
   ────────────                              ────────────

   4. дать все параметры              ◀───▶  5. не подтверждать без них
      задачи                                    (мысленная симуляция)


   7. спросить о препятствиях         ◀───▶  14. не молчать, если видишь
      до запуска процесса                       что правило нарушено


   13. тестировать в среде            ◀───▶  16. сказать отправителю, если
       получателя до публикации                   в твоей среде не работает
```

## Архетипы и специализации

Несколько правил-«родителей» порождают специализированные правила для разных носителей информации.

```
   1. Готовый к использованию вид
      │
      ├── 3.  Конкретные значения в ошибках (логи)
      ├── 4.  Все параметры в задаче (постановка)
      └── 15. Текст ссылки = информация (гипертекст)

   8. Добавляй контекст
      │
      └── будущие правила про сообщения,
          скриншоты, документы...

   9. Общий язык
      │
      ├── 10. Протокол (для повторяющихся взаимодействий)
      ├── 11. Стабильность формата (во времени)
      ├── 12. От слов к структуре (когда нужна однозначность)
      ├── 17. Канал приёма (согласование протокола перед передачей)
      └── 18. Эволюция контракта (согласование пересмотра по ходу)
```

## Жизненный цикл задачи

Если посмотреть на правила не по семействам, а по стадиям одной задачи — получится другая карта. Видно, на каких стадиях у нас правил много, а на каких — пусто.

```
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │ ПОСТАНОВКА  │──▶│   ПРИЁМ     │──▶│ ВЫПОЛНЕНИЕ  │──▶│  ПЕРЕДАЧА   │──▶│  ОБРАТНАЯ   │
  │             │   │             │   │             │   │  РЕЗУЛЬТАТА │   │   СВЯЗЬ     │
  │ отправитель │   │ исполнитель │   │ (часто внутри│  │ исполнитель │   │ получатель  │
  │  формирует  │   │  принимает  │   │ головы или   │  │  передаёт   │   │  оценивает  │
  │   задачу    │   │   задачу    │   │  системы)    │  │  результат  │   │  результат  │
  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘   └──────┬──────┘   └──────┬──────┘
         │                 │                 │                 │                 │
         ▼                 ▼                 ▼                 ▼                 ▼
   4, 6, 7, 8, 17         5, 14      9, 10, 11, 12, 18    1, 2, 3, 13, 15       16
   ──────────────         ──────     ──────────────────   ───────────────       ──
       плотно             средне        фон/системы          плотно           тонко
```

### По стадиям

- **Постановка.** 4 — все параметры; 6 — с запасом времени; 7 — спроси о препятствиях; 8 — контекст; 17 — согласуй канал приёма.
- **Приём.** 5 — не подтверждай вслепую; 14 — не молчи, если видишь грядущее нарушение.
- **Выполнение.** 9–12, 18 работают фоном: общий язык, протокол, стабильность формата, переход к структуре, эволюция контракта.
- **Передача результата.** 1 — готовый вид; 2 — не заставляй идти; 3 — конкретика; 13 — среда получателя; 15 — текст ссылки.
- **Обратная связь.** 16 — сообщи, если в твоей среде не работает.

### Где «дыры»

- **Внутри «выполнения»** — у нас нет правил про прогресс-апдейты, явные блокеры, «я застрял, нужна помощь». Это территория для будущих кейсов.
- **«Обратная связь»** держится на одном правиле (16). Не хватает правил про приёмку результата: как формулировать «не то / не работает», как закрывать задачу однозначно (а не «вроде ок, посмотрим»).
- **Между «приёмом» и «выполнением»** — переход не описан: что именно исполнитель должен сделать сразу после «ок», чтобы потом не было «забыл / не дошли руки».

### Что эта карта подсказывает

Большая часть правил живёт **на границах** — постановка и передача результата. Это разумно: на границах информация переходит из одной головы в другую, там и ломается. Но это же даёт смещение: середина задачи (выполнение) и конец (приёмка) у нас сейчас слабее. Это направление для следующих кейсов.

## Роли (расширение пары «отправитель ↔ получатель»)

Бинарная пара удобна как модель, но в реальной задаче ролей обычно 3–4. И их часто путают: одна голова держит две роли и не замечает, что одна из них «спит».

```
   ИНИЦИАТОР ───▶ ПОСТАНОВЩИК ───▶ ИСПОЛНИТЕЛЬ ───▶ ПОЛУЧАТЕЛЬ
     зачем            что               как             использует
                                                            │
                                                            ▼
                                                       НАБЛЮДАТЕЛЬ
                                                       (в курсе, но
                                                        не действует)
```

- **Инициатор** — у кого боль, нужда, интерес в результате.
- **Постановщик** — кто формулирует задачу для исполнителя.
- **Исполнитель** — кто делает работу.
- **Получатель** — кто использует результат.
- **Наблюдатель** — кому нужно быть в курсе (cc, аудит, контекст), но не действовать.

Роли часто совпадают, но **не всегда**:

| Сценарий | Кто кем |
|---|---|
| Вопрос в чате | спрашивающий = инициатор + постановщик + получатель; отвечающий = исполнитель; чат = наблюдатели |
| Корпоративная задача | продакт = инициатор; менеджер = постановщик; разработчик = исполнитель; конечный пользователь = получатель |
| Стройка / монтаж на режимном объекте | заказчик = инициатор + получатель; PM = постановщик; рабочие = исполнители; служба доступа = наблюдатель-привратник |

### Правила по переходам между ролями

```
   инициатор ─▶ постановщик       ──   (?) — у нас правил нет

   постановщик ─▶ исполнитель     ──   4, 6, 7, 8, 17   (со стороны постановщика)
                                       5, 14            (со стороны исполнителя)

   исполнитель ─▶ получатель      ──   1, 2, 3, 13, 15   (со стороны исполнителя)
                                       16                (со стороны получателя)

   наблюдатель ─▶ кто угодно      ──   (?) — у нас правил нет
```

### Где «дыры» с точки зрения ролей

- **Инициатор → постановщик.** Когда это разные люди, важно как инициатор передаёт «зачем» — иначе постановщик сформулирует правильное по форме, но не то по сути задание. Правил у нас пока нет.
- **Наблюдатель.** Когда вмешаться, когда молчать. Правило 14 («не молчи, если видишь нарушение») сейчас отнесено к получателю — на самом деле оно про **любого**, кто видит проблему. Возможно, его стоит обобщить.
- **Совмещение ролей в одной голове.** Самый частый источник ошибок — когда инициатор и постановщик это один человек: «зачем» не выговаривается, потому что внутри головы оно очевидно. Нужно отдельное правило: проговори «зачем» вслух, даже если сам себе ставишь задачу.

### Что это даёт

Ролевая карта показывает: наша бинарная пара была упрощением. Большая часть правил живёт на одной из двух границ (постановщик↔исполнитель и исполнитель↔получатель). Две другие границы — инициатор↔постановщик и наблюдатель↔все — пока пусты. Это новый угол для поиска кейсов.
